# План доработки мобильной CRM и управления портфолио Black Carp

Дата: 2026-07-15  
Режим: анализ и планирование, без изменения продуктового кода  
Целевая среда: Telegram WebApp, mobile-first  
Основные размеры проверки: 390×844, 430×932, 768×1024, 1024×900, 1440×900

## 1. Цель

Превратить текущую CRM из длинной адаптивной страницы со списком заявок в полноценное мобильное рабочее пространство мастера и добавить отсутствующее управление публичным портфолио.

Результат должен позволять мастеру с телефона:

1. быстро увидеть новые и требующие внимания заявки;
2. открыть одну заявку без прокрутки через весь список;
3. выполнить основные действия одной рукой;
4. безопасно вернуться к прежнему месту списка;
5. открыть раздел портфолио;
6. загрузить, отредактировать, упорядочить, опубликовать и снять с публикации работу;
7. сразу увидеть изменения в публичном разделе «Работы» сайта.

## 3. Подтвержденное текущее состояние

### 3.1. CRM

CRM реализует только работу с заявками:

- список, счетчики, поиск и фильтры;
- карточку заявки;
- смену статуса;
- заметки;
- планирование;
- вложения заявки;
- историю действий.

В CRM нет глобальной навигации по рабочим разделам. Верхняя панель содержит только бренд и кнопку «Обновить».

На ширине до 760 px `.workspace` становится обычным блочным документом. Список получает минимум в одну высоту экрана, а карточка добавляется ниже и тоже получает минимум в одну высоту экрана. Открытие заявки вызывает `scrollIntoView()`, а «К списку» прокручивает документ обратно вверх.

Это не модель мобильного приложения «список → карточка», а два полноразмерных экрана, физически поставленных друг под другом.

### 3.2. Портфолио

Управления портфолио в рабочей CRM нет.

Публичные работы сейчас:

- жестко записаны в `index.html` как восемь `article.work-slide`;
- используют четыре локальных медиафайла, часть изображений повторяется;
- не имеют сущности в SQLite;
- не имеют публичного или CRM API;
- не имеют статусов черновик/опубликовано/архив;
- не имеют управляемого порядка, обложки, alt-текста и признака публикации.

`editor-server.js` не решает задачу портфолио:

- это отдельный локальный HTML-редактор;
- он не подключен к текущему production-серверу;
- он не использует Telegram CRM-авторизацию;
- он сохраняет весь `index.html` целиком и создает резервные HTML-файлы;
- он не предоставляет модель портфолио, безопасную публикацию и аудит действий;
- используемые им `multer` и `cheerio` отсутствуют в текущих production dependencies.

Не развивать `editor-server.js` как админку портфолио. Новый модуль должен быть частью защищенной CRM и текущего backend.

## 5. Карта проблем

### P0 — блокирует нормальное мобильное использование

1. Список и карточка находятся в одном вертикальном документе.
2. После выбора заявки мастер сначала остается внутри высоты списка, затем перемещается к карточке ниже.
3. Возврат реализован прокруткой, а не переходом состояния.
4. Нет глобальной мобильной навигации между «Заявками» и «Портфолио».
5. Управление портфолио полностью отсутствует.
6. Нет гарантированного восстановления позиции, фильтра и поискового запроса после возврата из карточки.

### P1 — мешает ежедневной работе

7. Верхняя область слишком высокая для Telegram WebApp: Telegram chrome + topbar + крупный заголовок съедают значительную часть первого экрана.
8. Фильтры имеют текст 11 px и вертикальные отступы 5 px; удобная touch-зона 44×44 не обеспечена.
9. Метаданные строк и даты имеют 11 px, конкурируют между собой и плохо сканируются.
10. Счетчики показывают только три статуса и не объясняют, что требует внимания сейчас.
11. Ручное «Обновить» является главным способом получить свежие данные.
12. В карточке основные действия распределены по длинной странице, нет компактной фиксированной панели действий.
13. Смена статуса через select смешивает просмотр состояния и выполнение команды.
14. Состояние сохранения показывается далеко, в верхней панели, а не рядом с изменяемым блоком.
15. Выбранная заявка имеет только визуальный класс, но не получает полноценного семантического состояния выбора.

### P2 — качество и доступность

16. Весь список и карточка используются как крупные live-регионы и полностью перерисовываются.
17. Плотность, размеры текста и границы дают слабую иерархию на реальном телефоне.
18. Терминальные статусы продолжают выглядеть как редактируемый select с одним вариантом.
19. Пустые заметки и вложения выглядят как отсутствие интерфейса, а не оформленное пустое состояние.
20. Нет браузерных регрессионных тестов мобильной навигации и responsive-состояний.

## 6. Целевая информационная архитектура

### 6.1. Глобальный shell CRM

На мобильном CRM должна иметь три стабильных слоя:

1. компактный верхний бар;
2. один активный рабочий экран;
3. нижнюю навигацию по разделам.

Первый релиз навигации:

- `Заявки`;
- `Портфолио`.

Не добавлять пустые разделы «Календарь», «Клиенты» и «Настройки» только ради вида. Их можно подключить позже тем же навигационным контрактом.

Правила:

- одновременно виден только один раздел;
- внутри заявок одновременно виден либо список, либо карточка;
- нижняя навигация скрывается или меняется на панель действий только там, где это необходимо для безопасного редактирования;
- topbar должен учитывать `env(safe-area-inset-top)`;
- bottom nav должен учитывать `env(safe-area-inset-bottom)`;
- высота topbar ориентировочно 52–56 px без учета safe area;
- все основные цели касания не менее 44×44 px.

### 6.2. Навигационная модель

Состояния URL:

- `/crm?section=requests` — список заявок;
- `/crm?section=requests&request=BC...` — карточка;
- `/crm?section=portfolio` — список работ;
- `/crm?section=portfolio&item=<id>` — редактор работы;
- `/crm?section=portfolio&mode=new` — создание работы.

Обязательная синхронизация:

- browser history;
- Telegram BackButton;
- локальное состояние фильтров;
- позиция прокрутки списка;
- защита несохраненного черновика.

### 6.3. Экран списка заявок

Порядок элементов сверху вниз:

1. компактный заголовок `Заявки` и индикатор синхронизации;
2. компактная полоса внимания: `Новые`, `Нужен ответ`, `Сегодня`;
3. поиск;
4. горизонтально прокручиваемые filter chips в одну строку;
5. список карточек заявок.

Изменения строк:

- первая строка: имя или понятный fallback + индикатор внимания;
- вторая: зона, размер, статус;
- третья: контакт или последнее значимое событие;
- справа: контекстная дата с подписью (`создана`, `обновлена`, `сеанс`);
- статус нельзя кодировать только цветом;
- вся карточка — одна touch-цель не менее 56 px высотой;
- новые сообщения и ошибки доставки должны поднимать заявку вверх, когда backend получит модель внимания.

Поиск и фильтры не должны исчезать или сбрасываться после открытия карточки.

### 6.4. Экран карточки заявки

На мобильном порядок должен соответствовать рабочим действиям:

1. back + код заявки;
2. имя, контакт и текущий статус;
3. основные действия: связаться, изменить статус, назначить дату;
4. сообщения и заметки;
5. краткая анкета;
6. вложения;
7. полная история.

Требования:

- список не должен присутствовать выше карточки в DOM-потоке активного экрана;
- back возвращает прежнюю позицию списка без визуального скачка;
- Telegram BackButton выполняет то же действие;
- основные действия доступны в sticky action bar или в верхней рабочей зоне;
- изменение статуса — отдельная команда с подтверждением, а не мгновенная запись при выборе option;
- feedback сохранения локален для формы;
- несохраненная заметка не теряется при смене статуса, возврате и временном обновлении данных;
- терминальный статус показывается как read-only состояние с допустимыми correction actions, если backend их поддерживает.

## 7. Целевой модуль портфолио

### 7.1. MVP-возможности

Мастер должен уметь:

1. видеть опубликованные, черновые и архивные работы;
2. создать работу;
3. загрузить изображение;
4. заполнить название, подпись, alt-текст, зону, стиль и год;
5. выбрать обложку/позицию кадра;
6. сохранить черновик;
7. опубликовать;
8. снять с публикации;
9. изменить порядок;
10. архивировать;
11. открыть публичное представление работы.

Удаление файла навсегда не должно быть первым действием. MVP использует архивирование; физическое удаление — отдельное подтвержденное действие после проверки, что файл нигде не используется.

### 7.2. Модель данных

Добавить версионированную миграцию и таблицы.

`portfolio_items`:

| Поле | Назначение |
|---|---|
| `id TEXT PRIMARY KEY` | внутренний ID |
| `slug TEXT UNIQUE NOT NULL` | стабильный публичный идентификатор |
| `title TEXT NOT NULL` | название работы |
| `caption TEXT` | короткая подпись |
| `alt_text TEXT NOT NULL` | доступное описание изображения |
| `body_zone TEXT` | зона тела |
| `style TEXT` | стиль |
| `year INTEGER` | год |
| `status TEXT NOT NULL` | `draft`, `published`, `archived` |
| `sort_order INTEGER NOT NULL` | порядок публикации |
| `featured INTEGER NOT NULL DEFAULT 0` | избранная работа |
| `media_type TEXT NOT NULL` | на первом этапе `image` |
| `original_path TEXT NOT NULL` | исходник в приватном storage |
| `display_path TEXT NOT NULL` | оптимизированный публичный файл |
| `thumb_path TEXT NOT NULL` | миниатюра CRM |
| `width INTEGER` / `height INTEGER` | размеры результата |
| `focal_x REAL` / `focal_y REAL` | точка кадрирования 0…1 |
| `published_at TEXT` | время публикации |
| `created_at TEXT NOT NULL` | создание |
| `updated_at TEXT NOT NULL` | обновление |

`portfolio_activity`:

- `id`;
- `portfolio_item_id`;
- `master_telegram_id`;
- `event_type`;
- `payload_json`;
- `created_at`.

Индексы:

- `(status, sort_order)`;
- `slug` unique;
- `updated_at`.

Не хранить порядок только в DOM или JSON-файле. Не редактировать `index.html` при каждом изменении портфолио.

### 7.3. Хранение медиа

Целевая структура:

- приватные оригиналы: `uploads/portfolio/original/`;
- оптимизированные публичные изображения: `uploads/portfolio/public/`;
- миниатюры CRM: `uploads/portfolio/thumb/`.

Требования к загрузке:

- разрешить JPEG, PNG и WebP;
- проверять реальный MIME и сигнатуру, не только расширение;
- задать лимит размера и пикселей;
- нормализовать имя файла;
- удалять EXIF/GPS при обработке;
- создавать display и thumbnail варианты;
- не публиковать original path напрямую;
- при ошибке обработки удалять частично созданные файлы;
- при повторной отправке не создавать дубликат операции;
- storage должен использовать production bind mount, а не Git checkout.

Видео не включать в первый MVP управления портфолио. Добавлять отдельной фазой после стабильной загрузки и публикации изображений.

### 7.4. API

Защищенные CRM endpoints, все через существующий `requireCrmAuth`:

- `GET /api/crm/portfolio?status=&q=&cursor=`;
- `POST /api/crm/portfolio` — создать черновик и принять metadata;
- `POST /api/crm/portfolio/:id/media` — загрузить/заменить изображение;
- `GET /api/crm/portfolio/:id`;
- `PATCH /api/crm/portfolio/:id` — изменить metadata;
- `POST /api/crm/portfolio/:id/publish`;
- `POST /api/crm/portfolio/:id/unpublish`;
- `POST /api/crm/portfolio/:id/archive`;
- `POST /api/crm/portfolio/reorder` — сохранить полный валидированный порядок или атомарный move;
- `GET /api/crm/portfolio/:id/preview` — защищенный preview черновика.

Публичный endpoint:

- `GET /api/portfolio` — только `published`, отсортированные по `sort_order`, без original path и внутренних полей.

Контракты:

- все mutations принимают `operationId`;
- обновления используют `version` или `updatedAt` precondition;
- конфликт возвращает `409` и свежую версию;
- reorder выполняется одной транзакцией;
- publish запрещен без изображения, title и alt_text;
- публичный ответ имеет ETag или короткий cache contract с корректной инвалидизацией после публикации.

### 7.5. Интерфейс списка портфолио

Экран:

- заголовок `Портфолио`;
- primary CTA `Добавить работу`;
- chips `Опубликовано`, `Черновики`, `Архив`;
- компактная сетка 2 колонки на телефоне;
- карточка: thumbnail, название, статус, позиция, дата обновления;
- быстрые действия через меню, но публикация/архив требуют понятного подтверждения.

Порядок:

- drag-and-drop допустим как ускорение;
- обязательно дать доступные кнопки `Поднять` / `Опустить`;
- после сохранения показывать локальное подтверждение;
- на ошибке восстанавливать прежний порядок.

### 7.6. Редактор работы

Порядок:

1. изображение и состояние загрузки;
2. название;
3. подпись;
4. alt-текст с пояснением;
5. зона, стиль, год;
6. настройка focal point;
7. preview;
8. sticky actions `Сохранить` и `Опубликовать`.

Состояния:

- пустой новый черновик;
- загрузка;
- обработка изображения;
- upload error + retry;
- dirty draft;
- saving;
- saved;
- conflict;
- publish validation error;
- published;
- archived.

При уходе с dirty draft показать `Сохранить / Не сохранять / Остаться` и согласовать это поведение с Telegram BackButton.

## 8. Интеграция с публичным сайтом

Текущие восемь статических `work-slide` должны быть заменены данными из `GET /api/portfolio`.

Правила:

- публично показываются только `published`;
- порядок полностью определяется `sort_order`;
- счетчик `01 / NN` вычисляется по реальному числу работ;
- `alt` берется из обязательного `alt_text`;
- подпись строится из управляемых полей, а не из hardcoded HTML;
- загрузка использует responsive image variants;
- при временной ошибке API раздел не должен стать пустым: сохранить небольшой безопасный fallback или last-known published manifest;
- изменение черновика не меняет сайт;
- publish/unpublish должны менять сайт предсказуемо и атомарно.

Начальная миграция:

1. импортировать существующие уникальные работы как черновики;
2. не создавать восемь записей из повторяющихся картинок автоматически;
3. дать мастеру проверить title, caption и alt;
4. опубликовать подтвержденный набор;
5. только после этого удалить hardcoded source-of-truth из `index.html`.

## 10. Тестовая матрица для ИИ-агента

### Mobile navigation

- список → карточка → back восстанавливает scroll;
- deep link сразу открывает карточку;
- reload сохраняет понятное состояние;
- Telegram BackButton совпадает с UI back;
- dirty draft блокирует потерю данных;
- переключение `Заявки ↔ Портфолио` не оставляет два экрана видимыми.

### Requests

- empty/loading/error/auth/not-found;
- поиск и фильтры после возврата;
- delayed response A после B не возвращает карточку A;
- refresh не сбрасывает активную форму;
- touch targets не менее 44×44;
- нет horizontal overflow;
- выбранная строка доступна assistive technology;
- focus переходит в карточку и возвращается к строке.

### Portfolio API

- CRM auth required на каждом mutation;
- invalid MIME/extension mismatch;
- oversized file и decompression bomb limit;
- path traversal filename;
- duplicate operationId;
- stale version conflict;
- publish без title/alt/media;
- draft не попадает в public API;
- atomic reorder;
- unpublish исчезает из public API;
- archive не удаляет используемый файл;
- обработка upload откатывает частичные файлы при ошибке.

### Portfolio UI

- create draft;
- upload success/progress/error/retry;
- заполнение metadata;
- preview draft;
- save conflict;
- publish validation;
- accessible reorder кнопками;
- восстановление порядка на error;
- dirty guard;
- public preview после publish;
- empty portfolio state.

### Public works

- только published;
- правильный sort order;
- `01 / NN`;
- alt для каждого изображения;
- fallback при недоступном API;
- отсутствие layout shift сверх принятого порога;
- responsive image не загружает original без необходимости.

## 11. Definition of Done

Задача не считается выполненной только по `npm test` или API-тестам.

Обязательно:

1. `npm run check` проходит;
2. API/integration tests проходят;
3. browser tests проходят на указанной матрице;
4. свежие screenshots списка, detail, portfolio list/editor и public works сохранены;
5. нет horizontal overflow;
6. mobile list/detail являются разными состояниями, а не блоками одного длинного документа;
7. Telegram BackButton и dirty guard проверены;
8. touch targets и safe areas проверены;
9. portfolio draft не доступен публично;
10. publish/unpublish/reorder подтверждены на public site;
11. storage переживает перезапуск контейнера и не находится в Git checkout;
12. migration dry-run и backup/restore проверены;
13. физический smoke выполнен на Telegram iOS и Android;
14. никакая существующая booking/CRM авторизация не ослаблена.

## 12. Ограничения для исполняющего ИИ-агента

- Не начинать с визуального полирования. Сначала исправить shell и one-pane navigation.
- Не переписывать проект на React/другой framework без отдельного решения: текущий vanilla JS можно декомпозировать постепенно.
- Не использовать `editor-server.js` как production portfolio admin.
- Не сохранять изменения портфолио путем перезаписи `index.html`.
- Не хранить пользовательские upload-файлы в Git checkout.
- Не делать drag-and-drop единственным способом сортировки.
- Не публиковать original files и приватные storage paths.
- Не считать API tests доказательством мобильной готовности.
- Не менять production/deploy до прохождения локальной migration и browser matrix.
- Не трогать соседние сервисы или домены на общем сервере.
- Сохранять текущую Telegram initData авторизацию для всех CRM операций.
- Выполнять работу небольшими фазами; после каждой фазы сохранять screenshots, тесты и короткий progress checkpoint.
