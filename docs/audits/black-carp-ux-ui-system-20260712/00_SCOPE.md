# Scope

## Target

Текущий UX/UI системы Black Carp после production-доработки: публичный сайт, встроенная анкета записи и профиль заявки, Telegram-бот, CRM/WebApp мастера и связи между этими поверхностями.

## User goals

- Клиент понимает ценность мастера, просматривает работы и без потерь отправляет заявку.
- Клиент понимает, что произошло после отправки и как будет продолжено общение.
- Мастер своевременно замечает заявку, быстро понимает контекст и безопасно ведёт её до завершения.

## Included

- `index.html`, `styles.css`, `script.js`, медиаматериалы сайта.
- `server.js`: booking/status API, Telegram webhook/messages, CRM API и состояния.
- `crm/index.html`, `crm/styles.css`, `crm/app.js`.
- Мобильный и десктопный вид, основные, пустые, загрузочные, ошибочные и восстановительные состояния.
- Доступность, тексты, навигация, обратная связь, согласованность статусов и переходов.
- Текущие тесты и спецификации как заявленное поведение.

## Non-goals

- Исправление продуктового кода в этом проходе.
- Проверка реального production-деплоя, реальных Telegram-аккаунтов и физических устройств без отдельного доступа.
- Редизайн бренда с нуля и оценка качества временного портфолио-контента как самостоятельная арт-задача.
- `editor-server.js` и визуальный редактор не считаются текущей CRM: они не входят в production runtime `npm start`; учитываются только как потенциально конфликтующий legacy-контур.

## Roles

- Клиент сайта.
- Клиент, связавший заявку с Telegram.
- Мастер в Telegram и CRM WebApp.

## Verification

- Скриншоты создаются заново в текущем прогоне через in-app Browser.
- Локальный сервер запускается на изолированной БД в `/private/tmp`; внешние Telegram-запросы не отправляются.
- Кодовая проверка: точечное чтение, `npm test`, `npm run check`, API/DOM/keyboard/responsive checks.
- Для Telegram-бота без реального Telegram UI используются код, тестовый webhook и зафиксированные Bot API payloads; визуальная проверка будет явно отмечена как пробел.
