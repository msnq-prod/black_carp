# ТЗ на разработку MVP CRM Black Carp в Telegram WebApp

Дата: 2026-07-06

## 1. Цель

Сделать MVP CRM для мастера Black Carp внутри Telegram WebApp.

CRM должна заменить управление заявками через сообщения бота. Бот остается только транспортом уведомлений: сообщает мастеру о новой заявке и открывает WebApp по кнопке.

## 2. Текущая ситуация

Сейчас сайт отправляет анкету в `POST /api/booking/submit`, backend сохраняет заявку в SQLite и возвращает клиенту ссылку Telegram deep link.

Текущий поток:

```text
сайт -> backend -> клиент открывает Telegram -> /start CODE -> мастер получает заявку
```

Для новой CRM этот поток нужно изменить:

```text
сайт -> backend -> мастер сразу получает уведомление -> мастер открывает WebApp CRM
```

Клиентский переход в Telegram больше не должен быть обязательным условием доставки заявки мастеру.

## 3. Роли

### Клиент

Клиент заполняет анкету на сайте и оставляет контакт.

После отправки клиент видит экран успеха на сайте. Telegram открывать необязательно.

### Мастер

Мастер получает уведомление в Telegram-боте и открывает заявку в WebApp CRM.

Мастер может:

- смотреть новые заявки;
- открывать карточку заявки;
- менять статус;
- оставлять внутренние заметки;
- назначать дату консультации или сеанса;
- видеть историю изменений.

### Бот

Бот только:

- принимает webhook от Telegram;
- отправляет мастеру уведомления;
- открывает WebApp CRM через кнопку;
- при необходимости сообщает клиенту простые сервисные сообщения.

Бот не должен быть основным интерфейсом CRM.

## 4. MVP-объем

В MVP входит:

1. Изменение клиентской анкеты на сайте.
2. Сохранение заявки с контактом клиента.
3. Мгновенное уведомление мастера после создания заявки.
4. Telegram WebApp CRM для мастера.
5. Авторизация мастера через Telegram initData.
6. Список заявок.
7. Карточка заявки.
8. Смена статуса заявки.
9. Внутренние заметки мастера.
10. Базовое назначение даты/времени.
11. История действий по заявке.

В MVP не входит:

- полноценная десктопная CRM;
- несколько салонов;
- сложная ролевая модель;
- онлайн-оплата;
- автоматический календарь свободных слотов;
- клиентский личный кабинет;
- рассылки;
- интеграция с внешними CRM;
- аналитика продаж;
- миграция на PostgreSQL.

## 5. Клиентский сайт

### 5.1. Изменить финальный шаг анкеты

Текущую кнопку `Перейти в Telegram` заменить на:

```text
Отправить заявку
```

После успешной отправки показать экран:

```text
Заявка отправлена
Мастер получил вашу анкету и свяжется с вами.
```

Дополнительно можно показать:

- номер заявки;
- кнопку `Написать в Telegram`, но только как дополнительный канал, не как обязательный шаг;
- кнопку `Заполнить заново`.

### 5.2. Добавить контакт клиента

В анкету нужно добавить обязательный блок контакта:

- имя;
- предпочтительный канал связи;
- Telegram username или телефон.

Минимальные поля:

| Поле | Обязательность | Комментарий |
|---|---:|---|
| `clientName` | да | имя клиента |
| `contactType` | да | `telegram`, `phone`, `other` |
| `contactValue` | да | username, телефон или другой контакт |
| `contactComment` | нет | удобное время, пояснение |

Валидация:

- имя: 2-80 символов;
- контакт: 3-120 символов;
- телефон допускается в свободном формате, но без пустого значения;
- Telegram username допускается как `@username` или `username`.

## 6. Backend

### 6.1. Существующую платформу оставить

Для MVP оставить текущий стек:

- Node.js;
- Express;
- SQLite;
- локальные вложения в `uploads/booking/`;
- Telegram Bot API.

PostgreSQL пока не нужен. Его можно рассмотреть после появления нескольких мастеров, фильтров, календаря и отчетов.

### 6.2. Изменить `POST /api/booking/submit`

Новый результат endpoint:

1. Валидирует анкету.
2. Валидирует контакт клиента.
3. Сохраняет заявку.
4. Ставит статус `new`.
5. Сохраняет событие в историю.
6. Отправляет мастеру Telegram-уведомление.
7. Возвращает сайту номер заявки и статус отправки.

Ответ:

```json
{
  "ok": true,
  "requestId": "req_...",
  "publicCode": "BC123456",
  "status": "new",
  "masterNotified": true
}
```

Если Telegram недоступен:

```json
{
  "ok": true,
  "requestId": "req_...",
  "publicCode": "BC123456",
  "status": "new",
  "masterNotified": false
}
```

Важно: ошибка Telegram не должна ломать сохранение заявки.

### 6.3. Уведомление мастеру

Текст уведомления:

```text
Новая заявка Black Carp #BC123456

Клиент: Иван
Контакт: @username
Зона: Рука / Предплечье
Размер: M, около 15 см
Эскиз: нужна разработка

Открыть заявку в CRM
```

Кнопка:

```json
{
  "text": "Открыть заявку",
  "web_app": {
    "url": "https://black-carp.art/crm?request=BC123456"
  }
}
```

### 6.4. Telegram WebApp авторизация

CRM должна работать только внутри Telegram WebApp.

Frontend получает `Telegram.WebApp.initData` и отправляет его backend в каждом CRM-запросе:

```http
X-Telegram-Init-Data: <initData>
```

Backend обязан:

1. Проверить подпись `initData` через `BOT_TOKEN`.
2. Получить Telegram user id.
3. Проверить, что user id есть в списке мастеров.
4. Запретить доступ всем остальным.

Настройка:

```env
MASTER_TELEGRAM_IDS=406431910,1031438329
CRM_WEBAPP_URL=https://black-carp.art/crm
```

Если CRM открыта не из Telegram, показать экран:

```text
Откройте CRM из Telegram-бота Black Carp.
```

## 7. API CRM

Все endpoint ниже требуют валидный `X-Telegram-Init-Data`.

### 7.1. Текущий мастер

```http
GET /api/crm/me
```

Ответ:

```json
{
  "ok": true,
  "master": {
    "telegramId": 406431910,
    "name": "Nikita"
  }
}
```

### 7.2. Список заявок

```http
GET /api/crm/requests?status=new&limit=30&cursor=...
```

Фильтры MVP:

- `status`;
- `q` по номеру заявки, имени, контакту;
- `from`;
- `to`.

Ответ:

```json
{
  "ok": true,
  "items": [
    {
      "id": "req_...",
      "publicCode": "BC123456",
      "status": "new",
      "clientName": "Иван",
      "contactValue": "@username",
      "bodyZone": "Рука",
      "bodySubzone": "Предплечье",
      "sizeCm": 15,
      "hasSketch": false,
      "createdAt": "2026-07-06T00:00:00.000Z"
    }
  ],
  "nextCursor": null
}
```

### 7.3. Карточка заявки

```http
GET /api/crm/requests/:id
```

Возвращает:

- анкету;
- контакт;
- вложения;
- статус;
- заметки;
- историю действий;
- дату/время записи, если назначены.

### 7.4. Смена статуса

```http
PATCH /api/crm/requests/:id/status
```

Тело:

```json
{
  "status": "in_review"
}
```

Доступные статусы:

| Статус | Значение |
|---|---|
| `new` | новая заявка |
| `in_review` | мастер смотрит |
| `need_details` | нужно уточнение |
| `approved` | заявка одобрена |
| `scheduled` | назначена дата |
| `done` | работа завершена |
| `cancelled` | отменена |

### 7.5. Заметки мастера

```http
POST /api/crm/requests/:id/notes
```

Тело:

```json
{
  "text": "Клиент хочет тонкую графику, уточнить место."
}
```

### 7.6. Назначение даты

```http
PATCH /api/crm/requests/:id/schedule
```

Тело:

```json
{
  "scheduledAt": "2026-07-10T08:00:00.000Z",
  "durationMinutes": 180,
  "comment": "Консультация + эскиз"
}
```

## 8. База данных

### 8.1. Расширить `booking_requests`

Добавить поля:

```sql
client_name TEXT;
contact_type TEXT;
contact_value TEXT;
contact_comment TEXT;
scheduled_at TEXT;
duration_minutes INTEGER;
crm_updated_at TEXT;
```

### 8.2. Добавить `crm_notes`

```sql
CREATE TABLE crm_notes (
  id TEXT PRIMARY KEY,
  request_id TEXT NOT NULL REFERENCES booking_requests(id) ON DELETE CASCADE,
  master_telegram_id TEXT NOT NULL,
  text TEXT NOT NULL,
  created_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

### 8.3. Добавить `crm_activity`

```sql
CREATE TABLE crm_activity (
  id TEXT PRIMARY KEY,
  request_id TEXT NOT NULL REFERENCES booking_requests(id) ON DELETE CASCADE,
  actor_type TEXT NOT NULL,
  actor_id TEXT,
  event_type TEXT NOT NULL,
  payload_json TEXT,
  created_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

В `crm_activity` писать:

- создание заявки;
- отправку уведомления мастеру;
- открытие карточки;
- смену статуса;
- добавление заметки;
- назначение даты;
- ошибку Telegram-уведомления.

## 9. Интерфейс CRM WebApp

CRM делается mobile-first под Telegram viewport.

### 9.1. Экран списка

Содержимое:

- верхняя строка с названием `Black Carp CRM`;
- счетчики: `Новые`, `В работе`, `Назначены`;
- фильтры по статусам;
- список заявок карточками.

Карточка заявки:

- номер;
- имя клиента;
- контакт;
- зона;
- размер;
- статус;
- дата создания;
- индикатор вложений.

### 9.2. Карточка заявки

Блоки:

1. Клиент.
2. Контакт.
3. Анкета.
4. Вложения.
5. Статус.
6. Планирование.
7. Заметки.
8. История.

Основные действия:

- `Взять в работу`;
- `Нужно уточнение`;
- `Одобрить`;
- `Назначить дату`;
- `Завершить`;
- `Отменить`.

Контактные действия:

- открыть Telegram username;
- скопировать телефон или контакт;
- скопировать краткое резюме заявки.

### 9.3. Состояния интерфейса

Обязательные состояния:

- загрузка;
- пустой список;
- ошибка сети;
- нет доступа;
- заявка не найдена;
- сохранение статуса;
- сохранение заметки.

## 10. Безопасность

Обязательные требования:

1. Все `/api/crm/*` endpoint закрыты проверкой Telegram initData.
2. Доступ только Telegram ID из `MASTER_TELEGRAM_IDS`.
3. Нельзя доверять user id, присланному frontend без проверки подписи.
4. Вложения заявки не должны открываться через угадываемую публичную ссылку из CRM.
5. Максимальный размер вложений остается ограниченным.
6. Все изменения статусов и заметки пишутся в историю.
7. Ошибки Telegram API логируются, но не раскрывают `BOT_TOKEN`.

## 12. Критерии приемки

MVP считается готовым, если:

1. Клиент может отправить заявку без перехода в Telegram.
2. Backend сохраняет заявку, контакт и вложения.
3. Мастер получает Telegram-уведомление сразу после отправки заявки.
4. Кнопка в уведомлении открывает WebApp CRM.
5. CRM не открывается для посторонних Telegram ID.
6. Мастер видит список заявок.
7. Мастер открывает карточку заявки.
8. Мастер видит анкету, контакт и вложения.
9. Мастер меняет статус заявки.
10. Мастер добавляет внутреннюю заметку.
11. Мастер назначает дату и длительность.
12. Все действия попадают в историю.
13. Если Telegram API недоступен, заявка все равно сохраняется.
14. Проверки проходят локально и на production после деплоя.

## 13. Проверки

Минимальный набор:

```bash
node --check server.js
node --check script.js
```

Smoke-сценарии:

1. Отправить заявку с сайта.
2. Проверить запись в SQLite.
3. Проверить уведомление мастеру через Telegram stub.
4. Открыть CRM с валидным initData.
5. Проверить отказ с невалидным initData.
6. Сменить статус.
7. Добавить заметку.
8. Назначить дату.
9. Проверить историю действий.

Production-проверки:

1. `GET /health`.
2. Отправка реальной тестовой заявки.
3. Получение уведомления мастером.
4. Открытие WebApp из Telegram.
5. Проверка, что чужой Telegram ID не имеет доступа.

## 15. Главный принцип MVP

Не строить большую CRM заранее.

MVP должен закрыть один рабочий процесс:

```text
клиент отправил анкету -> мастер получил уведомление -> мастер открыл CRM -> мастер обработал заявку
```

Все остальное добавлять только после проверки этого потока в реальной работе.
