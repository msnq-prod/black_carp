Black Carp CRM MVP

Карта взаимодействий сайта, Telegram-бота, backend и CRM WebApp

Целевая схема: клиент отправляет заявку на сайте, backend сохраняет ее и сразу уведомляет мастера. Мастер открывает заявку в Telegram WebApp CRM, а бот остается только каналом уведомлений.

Главное изменение Клиент больше не обязан переходить в Telegram, чтобы мастер увидел заявку.
Основной интерфейс мастера Telegram WebApp CRM, открывается из кнопки в уведомлении бота.
Защита CRM Доступ только по валидному Telegram initData и разрешенным Telegram ID мастеров.
Клиент Сайт Backend / БД Telegram Bot API Мастер / CRM Аварийная ветка

Основной путь заявки

Слева направо: кто действует, куда уходят данные, что видят клиент и мастер.

Клиент

  • заполняет анкету на сайте;
  • оставляет контакт;
  • получает подтверждение без обязательного Telegram.

Сайт

  • собирает данные заявки;
  • показывает валидацию;
  • после submit показывает успех.

Backend

  • валидирует и сохраняет заявку;
  • пишет историю;
  • отдает CRM API и защищает доступ.

Бот

  • отправляет уведомления;
  • дает кнопку WebApp;
  • не является CRM-интерфейсом.

Мастер

  • видит очередь заявок;
  • меняет статусы;
  • добавляет заметки и дату.

Путь клиента

Открывает сайтвидит работы и переходит к записи.
Заполняет анкетуопыт, эскиз, зона, размер, идея, вложения.
Оставляет контактимя, Telegram или телефон.
Отправляет заявкувидит номер заявки и сообщение об успехе.

Путь мастера

Получает уведомлениебот присылает краткую карточку новой заявки.
Открывает CRMкнопка в Telegram запускает WebApp.
Разбирает заявкусмотрит контакт, анкету, вложения, историю.
Фиксирует решениестатус, заметка, дата консультации или сеанса.

Данные заявки

  • анкета клиента;
  • контакт;
  • вложения;
  • статус;
  • история действий.

CRM API

  • `GET /api/crm/me`;
  • `GET /api/crm/requests`;
  • `GET /api/crm/requests/:id`;
  • `PATCH status`;
  • `POST notes`;
  • `PATCH schedule`.

Контроль доступа

  • валидировать `initData`;
  • пускать только `MASTER_TELEGRAM_IDS`;
  • не доверять frontend user id;
  • логировать все изменения.

Жизненный цикл заявки

Статус Кто ставит Что означает Что видит мастер
new backend заявка только что пришла с сайта карточка в очереди новых
in_review мастер мастер открыл и разбирает заявку заявка в работе
need_details мастер нужны уточнения от клиента контакт и заметка, что спросить
approved мастер идея подходит, можно назначать время следующее действие — дата
scheduled мастер дата и длительность назначены заявка в расписании
done / cancelled мастер заявка закрыта архивная запись

Если Telegram API недоступен

  • заявка все равно сохраняется;
  • в истории пишется ошибка уведомления;
  • сайт показывает клиенту успех, если сохранение прошло;
  • мастер увидит заявку при открытии CRM.

Что не строим в MVP

  • платежи и предоплату;
  • автоматические свободные слоты;
  • клиентский личный кабинет;
  • сложные роли и несколько салонов;
  • отдельную desktop CRM.

Схема соответствует реализованной MVP-архитектуре: заявка сохраняется на сайте, мастер уведомляется через outbox и открывает защищённую Telegram WebApp CRM.
